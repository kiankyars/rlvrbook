# Chapter Archetypes

This book should not force every chapter into one shape. Use the lightest structure that fits the chapter's actual job.

## 1. Core Concept Chapter

Use for chapters that define a concept, distinction, or design problem.

Suggested sections:

- Chapter Map
- Core Terms
- Main Argument
- Design Patterns or Canonical Examples
- Limits, Failure Modes, or Open Questions

Good fits:

- The Verifier Lens
- Outcome Verifiers
- Process Verifiers
- Learned, Programmatic, and Hybrid Verifiers
- Turning Checks into Training Signal
- Search and Test-Time Verification
- Reward Hacking, Proxy Misspecification, and Verifier Robustness
- Faithfulness, Confidence, and What Verification Misses
- Open Problems and the Research Agenda

## 2. Domain Chapter

Use for chapters that compare verifier regimes across task families or applied settings.

Suggested sections:

- Chapter Map
- Domain Overview
- Verifier Regime
- Canonical Cases
- Comparative Lessons
- Open Questions

Good fits:

- Canonical Domains: Math, Code, and Formal Proof
- Long-Context, Multimodal, and Agentic RLVR

## 3. Appendix / Field Manual

Use for appendices, checklists, and operational reference material.

Suggested sections:

- Purpose
- What This Covers
- Practical Notes
- Checklist or Reference Material

## Rule of Thumb

Do not force sections like `Failure Modes` or `What the Verifier Misses` into every chapter. Include them when they clarify the chapter's real argument. Skip them when they only create structure without adding insight.
